## 起因

最近在 V2EX 看到一个帖子，楼主吐槽后端的同事在接口部署到测试环境后，遇到跨域报错竟然还在问前端“报什么错了”。楼主觉得无奈，认为这反映了部分后端开发者对跨域原理的认知缺失。

作为前端，开发时明明配个代理就能跑通，一发到生产环境就跨域。这种“开发能跑，发布就挂”的现象太常见了。要彻底解决协作中的沟通摩擦，我们需要跳出前端视角，重新理解“跨域”到底在防什么。

---

## 谁在拦截跨域请求？

很多刚接触全栈的开发者会误以为跨域是服务器的限制。实际上，跨域限制是**浏览器**的安全机制（同源策略），而不是服务器的缺陷。

浏览器的核心使命是保护用户安全。如果没有同源策略，恶意网站就能通过用户的浏览器直接向银行后台发送请求读取隐私数据。

当你从前端（A 域名）请求后端（B 域名）时，浏览器在发送请求前会产生顾虑：A 信任 B 吗？B 允许 A 读取数据吗？

CORS（跨域资源共享）本质上是服务器向浏览器发出的“担保声明”。后端通过响应头明确告诉浏览器：“我是 B，我认识 A，我允许它来调用我。”浏览器拿到这个证明，才会放行请求。

## 为什么本地能跑，生产不行？

在开发环境，你在 Vite 或 Webpack 里配置了 Proxy。

你的页面运行在 `localhost:5173`。代码发起 `/api/user` 请求，由于没写具体域名，请求默认发往当前的 `localhost:5173`。浏览器一看，协议、域名、端口完全一致，不用跨域核验，直接放行。

本地的 Vite 代理服务器（Node.js）收到这个请求后，将其转发给真实的后端 `localhost:3000`。

关键逻辑在这里：**服务器与服务器之间是可以随意通信的。** Node.js 不受浏览器同源策略的限制。它直接去 3000 端口把数据拿回来，再传回给浏览器。浏览器全程被蒙在鼓里，以为数据就是 5173 给的。

到了生产环境，前端变成静态文件跑在服务器上。如果架构上没做代理，浏览器只能直接向真实的后端 API 域名发请求。域名不同，瞬间触发 CORS 检查。如果后端没配置响应头，请求当场被拦截。

## 为什么服务器之间不用防跨域？

既然浏览器查得这么严，为什么服务器之间互相调接口就“随便通”？这听起来似乎很不合逻辑。

其实，信任是“规划”出来的，不是“防”出来的。

浏览器的环境极其复杂，运行着各种不知名的第三方脚本，需要“一刀切”的保姆式机制。而服务器架构是你自己控制的。如果你连自己写的后端组件都不信任，系统就没法运转了。

服务器之间的防御在更底层完成：

- **网络层白名单**：服务通常运行在 VPC（私有网络）内。通过安全组配置，只放行特定 IP。不在白名单里的机器，连握手的机会都没有。
- **应用层鉴权**：互相调用依靠密钥（Token 或 API Key）。没有正确的 Authorization 凭据直接返回 401。这比单纯校验域名要安全得多。

同源策略只是浏览器的第一道门。剩下的复杂验证，后端自己有一套极其严密的专业“门禁系统”。

## 从被动求助到架构参与

下次再遇到生产环境的跨域报错，不用觉得自己没权限改后端代码就束手无策了。

你只需要学会引导：

第一，**明确指出问题源头**。直接截图浏览器的报错日志：`No 'Access-Control-Allow-Origin' header is present`。告诉后端，这是浏览器拦截了请求，因为响应头缺少跨域权限声明。

第二，**直接给方案**。你可以提供一段标准的 Nginx 或中间件配置代码。通常你只要说一句“在 Nginx 的 api location 里加上跨域 Header”，事情就能解决大半。

第三，**推动同源部署**。这是消灭跨域的终极方案。建议运维在 Nginx 做反向代理，把访问 `/api/` 的流量转发给后端。这样前端页面和后端接口共用一个域名和端口，在浏览器眼里就是同源请求。

理解了网络请求链路的底层逻辑，你就不再只是一个被动发请求的前端，而是能从系统架构视角解决问题的开发者。
